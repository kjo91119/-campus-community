# 08. 인증, 익명성, 운영 정책 초안

문서 목적: 학교 인증, 학생증 보조 인증, 익명성, 신고/차단, 운영 기준을 MVP 수준으로 정의해 신뢰와 안전의 균형을 맞춘다.

## 가정 (Assumptions)

- 완전 비인증 익명 커뮤니티로는 가지 않는다.
- 공개 노출은 닉네임 중심이지만, 내부적으로는 인증된 사용자여야 한다.
- 운영 인력은 초기에 많지 않으므로 정책은 단순하고 실행 가능해야 한다.

## 인증 방식 초안

### 1. 학교 이메일 인증

기본 원칙:

- 가능한 경우 학교 이메일 인증을 우선 사용한다.
- 지원 학교 목록과 이메일 도메인을 매핑한다.

흐름:

1. 학교 이메일 입력
2. 도메인 검증
3. 인증 메일 또는 OTP 발송
4. 확인 완료 시 학교 인증 성공
5. 온보딩으로 이동

장점:

- 자동화가 쉽다
- 운영 리소스가 적게 든다
- 신뢰도가 높다

한계:

- 학교 이메일이 없는 학교 또는 사용 불가 사례 존재

### 2. 학생증 수동 인증

기본 원칙:

- 학교 이메일 인증이 불가능한 사용자만 보조 플로우로 허용한다.

흐름:

1. 학생증 제출 가이드 노출
2. 필수 정보만 보이도록 마스킹 안내
3. 이미지 업로드
4. 운영 검토
5. 승인 또는 반려

운영 포인트:

- 주민번호 등 과도한 민감정보는 가리도록 안내
- 검토 기준을 짧고 명확하게 문서화
- 반려 시 이유를 선택형으로 제공

## 인증 상태별 권한 매트릭스

| 상태 | 홈/모집 읽기 | 학교 보드 읽기 | 글/댓글/모집 작성 | 인증 재시도 | 비고 |
| --- | --- | --- | --- | --- | --- |
| `unverified` | 불가 | 불가 | 불가 | 가능 | 소개, 정책, 인증 진입 화면만 허용 |
| `pending` | 불가 | 불가 | 불가 | 가능 | 수동 인증 검토 상태 확인만 허용 |
| `verified` | 가능 | 같은 학교만 가능 | 가능 | 불필요 | 정상 사용자 |
| `rejected` | 불가 | 불가 | 불가 | 가능 | 반려 사유 확인 후 재제출 가능 |
| `restricted` | 제한적 가능 | 제한적 또는 불가 | 불가 | 불가 | 운영 제재에 따른 읽기 전용 상태 |
| `banned` | 불가 | 불가 | 불가 | 이의제기만 가능 | 계정 정지 |

권한 원칙:

- MVP에서는 "인증 전 읽기 허용"보다 "인증 후 커뮤니티 접근"을 더 강하게 유지한다.
- 학교 보드는 `verified` 상태와 학교 일치가 동시에 충족되어야 한다.
- `restricted` 사용자의 읽기 범위는 제재 사유에 따라 운영자가 조정하되, 기본값은 쓰기 전면 차단이다.

## 익명성 설계 원칙

### 기본 원칙

- 인증은 강하게, 공개 노출은 가볍게
- 실명은 노출하지 않는다
- 공개 기본값은 닉네임 기반이다

### 화면별 노출 정책

| 공간 | 공개 노출 기본값 | 내부 신뢰 정보 |
| --- | --- | --- |
| 통합 홈 | 닉네임, 전공군 정도만 노출 | 학교 인증 완료 여부 보유 |
| 전공군 피드 | 닉네임, 전공군 | 학교 정보는 기본 비노출 |
| 학교 보드 | 닉네임 또는 학교 내 익명 닉네임 | 같은 학교 인증 여부 필수 |
| 프로필 | 닉네임, 전공군, 일부 활동 정보 | 학교명 공개는 옵션 또는 제한 |

### 추천 정책

- 공개 프로필에는 실명, 학번을 노출하지 않는다.
- 학교명은 기본 비노출 또는 매우 제한적으로만 노출한다.
- 대신 "학교 인증 완료" 뱃지나 상태만 활용한다.

## 닉네임 / 학교명 표시 정책

### 닉네임

- 서비스 전반의 기본 공개 식별자
- 정책 위반 또는 혼동 유발 닉네임 제한
- 과도한 빈도 변경은 제한

### 학교명

- 운영과 권한 판정에는 사용
- 공개 노출은 최소화
- 학교 보드에서는 같은 학교 사용자에게만 맥락이 드러나면 충분

## 신고 / 차단 / 제재 기준

### 신고 대상

- 게시글
- 댓글/대댓글
- 모집글
- 사용자 프로필 또는 닉네임

### 신고 사유 예시

- 욕설/비방
- 혐오/차별
- 음란/부적절한 내용
- 광고/홍보/도배
- 허위 정보
- 사칭
- 모집 사기 의심

### 차단 정책

- 사용자는 다른 사용자를 개별 차단할 수 있다.
- 차단 후 해당 사용자의 글과 댓글은 기본적으로 숨긴다.
- 차단은 개인화 기능이며 제재와는 별개다.

### 제재 정책

- 1차: 경고 또는 특정 콘텐츠 블라인드
- 2차: 일정 기간 글쓰기/댓글 제한
- 3차: 계정 정지 또는 영구 제재

운영은 누적 신고 수만으로 자동 정지하지 않고, 운영 검토를 거친다.

## 악성 유저 대응 정책

초기에 꼭 필요한 대응 시나리오:

- 도배
- 반복 허위 모집
- 성희롱/혐오 표현
- 전공군/학교 간 분쟁 유도
- 개인정보 노출

대응 원칙:

- 콘텐츠 숨김과 계정 제재를 분리한다.
- 학교 인증이 되어 있어도 면책되지 않는다.
- 허위 신고 남용 사용자도 관리 대상에 포함한다.

## 운영자 검토가 필요한 항목

- 학생증 수동 인증 승인/반려
- 신고 누적 콘텐츠의 최종 판정
- 제재 이의제기
- 사칭 또는 허위 모집 의심 사례
- 학교명/전공군 매핑 예외 케이스

## MVP 운영 실행안

### 운영 역할

- 1차 검토자: 수동 인증과 일반 신고를 확인하는 담당자 1명
- 백업 검토자: 주말 또는 부재 시 대체 가능한 담당자 1명
- 제품/운영 책임자: 제재 수위와 예외 케이스 최종 판단

### 운영 도구

- 사용자 상태와 인증 요청: Supabase `profiles`, `verifications`
- 증빙 파일: Supabase Storage의 비공개 버킷
- 검토 기록: 운영 체크리스트 문서 또는 공유 시트 1개

초기에는 별도 관리자 웹 없이 위 세 가지로 운영한다.

### 수동 인증 SLA

| 단계 | 기준 |
| --- | --- |
| 접수 확인 | 영업일 기준 24시간 이내 |
| 최종 승인/반려 | 영업일 기준 72시간 이내 |
| 보완 요청 | 사진 불량 등 재제출 필요 시 1회 요청 |

### 신고 처리 SLA

| 유형 | 예시 | 기준 |
| --- | --- | --- |
| 긴급 | 개인정보 노출, 성희롱, 위협, 명백한 사칭 | 접수 후 6시간 이내 임시 숨김 또는 조치 |
| 일반 | 욕설, 도배, 허위 모집 의심, 분쟁 유도 | 영업일 기준 24시간 이내 1차 확인, 72시간 이내 처리 |

### 증빙 자료 보관 기준

- 수동 인증 증빙은 최종 승인 또는 반려 후 7일 내 삭제를 기본값으로 한다.
- 이의제기 또는 분쟁이 열린 경우에만 최대 30일까지 보관한다.
- 승인에 필요한 최소 정보만 확인하고, 장기 보관을 기본 정책으로 두지 않는다.

### 이의제기 기준

- 반려 또는 제재된 사용자는 7일 이내 1회 이의제기를 제출할 수 있다.
- 이의제기가 접수되면 백업 검토자 또는 책임자가 재검토한다.
- 동일 사안 반복 이의제기는 운영자 판단으로 종료할 수 있다.

## 개인정보 최소 수집 원칙

- 실명은 수집하지 않거나 최소화한다.
- 학번은 기본 수집 대상에서 제외한다.
- 학생증 수동 인증 자료는 검토 후 최소 보관 원칙을 둔다.
- 프로필 공개 정보는 닉네임, 전공군 정도로 최소화한다.

## 익명성과 신뢰의 균형

이 서비스는 완전 익명 커뮤니티도 아니고, 실명 커뮤니티도 아니다. 가장 적절한 균형은 아래와 같다.

- 내부 신뢰: 학교 인증으로 확보
- 외부 노출: 닉네임 기반으로 완화
- 민감한 학교 정보: 권한 판단에만 사용
- 학교 보드: 같은 학교 내에서만 제한적으로 소속감 활용

즉 사용자는 "인증된 사람들끼리 대화한다"는 안심을 가지면서도, 공개적으로는 과도한 신상 노출 없이 질문할 수 있어야 한다.

## 초기 운영 정책에서 꼭 필요한 최소 세트

1. 인증 정책
2. 닉네임 정책
3. 신고 사유 체계
4. 차단 기능
5. 학생증 수동 인증 검토 기준
6. 글쓰기/댓글 제재 기준
7. 개인정보 최소 수집 원칙
8. 수동 인증 및 신고 처리 SLA
9. 증빙 자료 삭제 기준

## 결론

MVP에서 가장 중요한 것은 "신뢰는 확보하되 부담은 낮추는 것"이다. 학교 인증을 통해 최소 신뢰를 만들고, 공개 노출은 닉네임 중심으로 설계하며, 신고/차단/제재 정책을 일찍 정의해야 커뮤니티가 빠르게 무너지지 않는다.

# 11. 개발 로드맵

문서 목적: 내일부터 실제 개발을 시작할 수 있도록 Phase별 목표, 구현 범위, 의존성, 리스크를 정리한다.

## 가정 (Assumptions)

- 현재는 문서화 완료 직후의 상태다.
- 프런트엔드 앱부터 먼저 만들고, Supabase 연동은 그에 맞춰 진행한다.
- 운영자 도구는 MVP 범위 밖이므로 최소 수동 운영을 전제로 한다.

## Phase 0. 기획 고정

목표:

- 제품 정의, MVP 범위, IA, 데이터 구조를 고정한다.

항목:

- 문서 정리 완료
- 용어 통일
- 화면 우선순위 확정
- 데이터 모델 초안 확정

선행조건:

- 없음

리스크:

- 문서 없이 바로 구현하면 방향이 흔들림

## Phase 1. 인증과 앱 골격

목표:

- 사용자가 앱에 들어와 인증을 완료하고 홈에 진입할 수 있게 한다.

항목:

- 인증 진입 화면
- 학교 이메일 인증 플로우
- 학생증 수동 인증 플로우
- 온보딩
- 사용자 세션 상태 관리
- 기본 탭 구조 정리

선행조건:

- 대학 도메인 정책 정리
- 프로필/인증 테이블 설계

리스크:

- 인증 플로우가 무거우면 초반 이탈률이 높아짐

## Phase 2. 커뮤니티 핵심 루프

목표:

- 읽기, 작성, 댓글의 최소 커뮤니티 루프를 만든다.

항목:

- 통합 홈 피드
- 전공군 필터
- 글 상세
- 댓글/대댓글
- 글쓰기
- 학교 보드

선행조건:

- posts/comments/boards 데이터 구조
- RLS 기본 정책

리스크:

- 홈과 홈 내부 전공군 집중 보기의 역할이 겹치면 UX가 혼란스러움

## Phase 3. 모집과 운영 안전장치

목표:

- 반복 방문 이유를 강화하고 운영 리스크를 줄인다.

항목:

- 모집 리스트/상세/작성
- 신고
- 차단
- 기본 제재 플로우

선행조건:

- recruitments, reports, blocks 설계

리스크:

- 모집 악용과 허위 글 관리 필요

## Phase 4. 측정과 안정화

목표:

- 지표 수집과 베타 안정화를 진행한다.

항목:

- analytics 이벤트 삽입
- 빈 상태 문구 보강
- 정책 화면 정리
- 버그 수정
- 베타 테스트

선행조건:

- 주요 화면과 핵심 플로우 완료

리스크:

- 이벤트 과수집 또는 누락
- 운영 프로세스 미정으로 인한 베타 혼선

## 주차별 초안 일정

아래 일정은 "최소 6주 기준의 보수적 초안"이다. 4주 베타를 전제로 잡기보다, 운영 정책과 콘텐츠 시딩까지 닫는 데 필요한 시간을 반영했다.

### 1주차

- 네비게이션, 모집 모델, 권한 매트릭스 확정
- Supabase 테이블과 RLS 초안 고정
- 운영 SLA와 증빙 보관 기준 확정

### 2주차

- 인증 진입, 학교 이메일 인증, 수동 인증 상태 화면
- 온보딩
- 세션 및 사용자 상태 구조 만들기

### 3주차

- 통합 홈
- 전공군 필터
- 글 상세
- 글쓰기
- 댓글/대댓글

### 4주차

- 학교 보드
- 기본 프로필/설정

### 5주차

- 모집 기능
- 신고/차단
- 운영 처리 플로우 점검

### 6주차

- 분석 이벤트
- 정책 화면 정리
- 초기 콘텐츠 시딩
- QA
- 베타 릴리스 게이트 점검

## 어떤 순서로 만들어야 가장 안전한가

1. 인증과 사용자 상태
2. 데이터 모델과 권한
3. 통합 홈
4. 글 상세와 댓글
5. 글쓰기
6. 학교 보드
7. 모집
8. 신고/차단
9. 지표 수집

이 순서를 권장하는 이유는 아래와 같다.

- 인증과 권한이 먼저 잡혀야 학교 보드와 작성 권한을 안전하게 처리할 수 있다.
- 홈과 글 상세가 먼저 있어야 사용자가 읽을 것이 생긴다.
- 모집은 핵심이지만 커뮤니티 기본 루프 뒤에 붙이는 편이 안전하다.

## 주요 의존성

- `profiles`, `verifications`, `boards`, `posts`, `comments`가 먼저
- 이후 `recruitments`, `reports`, `blocks`, `notifications`

## 예상 리스크와 대응

| 리스크 | 설명 | 대응 |
| --- | --- | --- |
| 인증 마찰 | 학교 이메일 미지원 사례 | 수동 인증 보조 플로우 유지 |
| 빈 피드 | 초기 콘텐츠 부족 | 통합 홈, 통합 모집, 시딩 콘텐츠 준비 |
| 운영 부담 | 신고/수동 인증 처리 | 초기 정책 단순화, 검토 대상 명확화 |
| 범위 확장 | 알림, 채팅, 소셜 기능 욕심 | MVP 제외 원칙 고정 |

## 결론

이 로드맵의 핵심은 "인증 -> 읽기 -> 쓰기 -> 모집 -> 안전" 순서다. 기능을 넓히는 대신 핵심 루프를 먼저 닫아야, 4개 전공군 동시 런칭 구조에서도 흔들리지 않고 MVP를 검증할 수 있다.

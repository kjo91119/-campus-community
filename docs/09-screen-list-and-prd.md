# 09. 화면 목록과 간단 PRD

문서 목적: MVP에서 실제로 필요한 화면을 정리하고, 각 화면의 목적, 핵심 컴포넌트, 필요한 데이터, 상태 변화를 구현 관점에서 빠르게 파악할 수 있게 만든다.

## 가정 (Assumptions)

- MVP는 핵심 루프 검증이 목표이므로 화면 수를 최소화한다.
- 하나의 화면에서 너무 많은 역할을 맡기기보다, 핵심 흐름이 명확한 구성으로 간다.

## 화면 우선순위 기준

1. 인증과 온보딩에 직접 필요한가
2. 첫 방문자의 정보 탐색에 직접 필요한가
3. 글/댓글/모집 같은 핵심 상호작용에 직접 필요한가
4. 운영 안전성을 보장하는가

## MVP 화면 목록

| 우선순위 | 화면 | 목적 |
| --- | --- | --- |
| P0 | 시작/인증 진입 | 서비스 가치 제안과 로그인 시작점 |
| P0 | 학교 이메일 인증 | 기본 인증 플로우 |
| P0 | 학생증 수동 인증 제출/상태 | 예외 인증 처리 |
| P0 | 온보딩 | 닉네임, 전공군, 정책 인지 |
| P0 | 통합 홈 | 첫 진입 메인 피드 |
| P0 | 글 상세 | 읽기와 댓글 상호작용 중심 |
| P0 | 글쓰기 | 일반글/질문글/모집글 작성 |
| P1 | 홈 내 전공군 집중 보기 상태 | 전공군 중심 탐색을 홈 안에서 처리 |
| P1 | 모집 리스트 | 모집 탐색 허브 |
| P1 | 모집 상세 | 모집 정보 확인 및 참여 |
| P1 | 학교 보드 | 학교 한정 질문과 소통 |
| P1 | 프로필/설정 | 내 정보와 인증 상태 관리 |
| P1 | 신고/차단 액션 시트 | 안전 기능 최소 세트 |

## 화면별 요구사항

### 1. 시작/인증 진입 화면

목적:

- 서비스 정체성을 짧게 전달하고 인증 플로우로 연결한다.

핵심 컴포넌트:

- 한 줄 가치 제안
- 학교 이메일 입력 버튼
- 수동 인증 안내 링크
- 이용 규칙 요약

필요 데이터:

- 대학 도메인 지원 여부

상태 변화:

- 미인증
- 인증 진행 중
- 인증 실패

### 2. 학교 이메일 인증 화면

목적:

- 가장 기본적인 인증 완료

핵심 컴포넌트:

- 이메일 입력폼
- 인증 메일/OTP 전송
- 재전송 버튼
- 오류 메시지

필요 데이터:

- 대학 이메일 도메인 목록
- 인증 토큰 상태

상태 변화:

- 입력 전
- 전송 중
- 대기
- 성공
- 실패

### 3. 학생증 수동 인증 제출/상태 화면

목적:

- 이메일 인증이 어려운 사용자의 예외 처리

핵심 컴포넌트:

- 업로드 가이드
- 이미지 업로드
- 제출 상태 배지
- 반려 사유 표시

필요 데이터:

- 인증 요청 상태
- 증빙 파일 경로

상태 변화:

- 미제출
- 검토 중
- 승인
- 반려

### 4. 온보딩 화면

목적:

- 닉네임과 전공군을 확정하고 첫 피드 진입 준비

핵심 컴포넌트:

- 닉네임 입력
- 전공군 선택 카드
- 정책 체크

필요 데이터:

- major_groups
- 닉네임 중복 검사 결과

상태 변화:

- 입력 중
- 유효성 실패
- 완료

### 5. 통합 홈 화면

목적:

- 앱의 핵심 탐색 허브

핵심 컴포넌트:

- 전공군 필터 칩
- 카테고리 칩
- 인기글 섹션
- 최신글 섹션
- 모집글 미리보기
- 빠른 글쓰기 CTA

필요 데이터:

- posts
- recruitments
- 사용자 전공군
- 차단 목록

상태 변화:

- 기본 전체 피드
- 전공군 필터 적용
- 카테고리 적용
- 빈 상태

### 6. 홈 내 전공군 집중 보기 상태

목적:

- 특정 전공군에 집중된 탐색을 홈 안에서 처리한다.

핵심 컴포넌트:

- 전공군 헤더
- 카테고리 탭
- 글 리스트
- 전체 홈으로 돌아가는 상태 전환

필요 데이터:

- 선택된 major_group 기준 posts

상태 변화:

- 전공군 변경
- 카테고리 변경
- 빈 상태

### 7. 글 상세 화면

목적:

- 읽기, 댓글, 신고, 차단의 중심 화면

핵심 컴포넌트:

- 글 헤더
- 본문
- 댓글 리스트
- 댓글 입력창
- 더보기 메뉴

필요 데이터:

- post
- comments
- author profile summary

상태 변화:

- 댓글 작성
- 댓글 삭제
- 신고 처리
- 차단 후 목록 갱신

### 8. 글쓰기 화면

목적:

- 일반글, 질문글, 모집글 작성

핵심 컴포넌트:

- 작성 위치 선택
- 글 유형 선택
- 제목/본문 입력
- 카테고리 선택
- 게시 버튼

필요 데이터:

- boards
- major_groups
- 사용자 인증/권한 상태

상태 변화:

- 입력 중
- 유효성 실패
- 업로드 중
- 완료

### 9. 모집 리스트 화면

목적:

- 모집글만 빠르게 탐색

핵심 컴포넌트:

- 모집 유형 필터
- 전공군 필터
- 마감 상태 배지

필요 데이터:

- recruitments joined with posts

상태 변화:

- 전체/필터
- 마감 포함 여부
- 빈 상태

### 10. 모집 상세 화면

목적:

- 모집 내용을 읽고 참여 의사를 남긴다

핵심 컴포넌트:

- 모집 개요
- 본문
- 참여 의사 댓글 CTA
- 작성자 정보 요약

필요 데이터:

- recruitment
- linked post
- comment count

상태 변화:

- 참여 댓글 작성 전
- 참여 댓글 작성 후
- 마감

### 11. 학교 보드 화면

목적:

- 같은 학교 학생만 보는 제한형 소통 공간

핵심 컴포넌트:

- 학교명 헤더
- 학교 보드 피드
- 학교 한정 글쓰기

필요 데이터:

- current user university
- university board posts

상태 변화:

- 인증 미완료 접근
- 정상 접근
- 빈 상태

### 12. 프로필/설정 화면

목적:

- 내 상태와 기본 설정 확인

핵심 컴포넌트:

- 닉네임
- 전공군
- 학교 인증 상태
- 차단 목록 진입
- 운영 정책/문의 진입

필요 데이터:

- profile
- verification summary

상태 변화:

- 인증 상태 변경
- 닉네임 변경 요청

### 13. 신고/차단 액션 시트

목적:

- 불건전 사용자와 콘텐츠에 즉시 대응

핵심 컴포넌트:

- 신고 사유 선택
- 차단 버튼
- 확인 UI

구현 메모:

- 초기 구현은 상세 화면 안의 inline safety card로 시작해도 된다.
- 이후 필요 시 action sheet + 확인 모달 조합으로 정리한다.

필요 데이터:

- 대상 엔터티 정보

상태 변화:

- 신고 제출
- 차단 완료

## 처음 구현할 화면 순서

1. 시작/인증 진입
2. 학교 이메일 인증
3. 학생증 수동 인증 제출/상태
4. 온보딩
5. 통합 홈
6. 글 상세
7. 글쓰기
8. 홈 내 전공군 집중 보기 상태
9. 학교 보드
10. 모집 리스트/상세
11. 프로필/설정
12. 신고/차단 액션 시트

## PRD 관점 핵심 메모

- 홈은 반드시 "첫 방문자가 읽을 것이 보이는 화면"이어야 한다.
- 별도 전공 탭 없이도 홈 필터만으로 핵심 전공 탐색이 가능해야 한다.
- 학교 보드는 있어야 하지만, 제품 중심 화면이 되어서는 안 된다.
- 모집은 리스트와 상세만 있어도 MVP 검증이 가능하다.
- 알림은 있으면 좋지만, MVP에서 우선순위를 낮춰야 한다.

## 결론

화면 설계는 많아 보이지만 실제 MVP 핵심은 인증, 홈, 글 상세, 글쓰기, 모집, 학교 보드 정도로 압축된다. 이 순서를 잘 지키면 전공 네트워크의 핵심 가치를 빠르게 검증할 수 있다.

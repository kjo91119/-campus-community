# 32. Report Block Safety Review Brief

## 목적

신고/차단 최소 골격을 `posts`, `comments`, `recruitments` 읽기 경로와 프로필 화면에 붙인 단계를 교차검증한다. 이번 단계의 핵심은 운영 큐나 서버 제재까지 넓히지 않고, 로컬 MVP 기준으로 신고 기록과 차단 필터링 경계를 먼저 안전하게 닫는 것이다.

## 이번 단계에서 검증할 것

1. 글 상세와 모집 상세에서 신고 사유 선택, 콘텐츠 신고, 작성자 차단이 동작하는지 확인
2. 차단 후 해당 작성자의 글/댓글/모집글이 기본적으로 숨겨지는지 확인
3. 차단 목록이 프로필 화면에 노출되고 차단 해제가 가능한지 확인
4. 신고/차단이 아직 로컬 데모 상태임을 코드와 문구에서 분명히 드러내는지 확인
5. 이번 단계가 운영 제재, 오프라인 outbox, 서버 신고 큐까지 과범위하게 넓어지지 않았는지 확인

## 리뷰어에게 요청할 체크 포인트

### A. 신고/차단 경계

- 신고 대상 타입이 현재 화면 범위와 맞게 최소 세트로 정리되어 있는가
- 차단이 개인화 기능으로만 동작하고, 운영 제재와 혼동되지 않는가
- 신고/차단 메시지가 로컬 MVP 단계라는 점을 숨기지 않는가

### B. 읽기 경로 반영

- 차단한 사용자의 글/댓글/모집글이 목록과 상세에서 기본적으로 숨겨지는가
- 차단 직후 현재 화면이 꼬이지 않도록 안전하게 이동하거나 갱신되는가
- 프로필의 차단 목록과 provider 상태가 같은 기준을 쓰는가

### C. 범위 관리

- 이번 단계에서 신고 서버 저장, 운영자 검토 큐, 자동 제재, outbox sync 같은 다음 단계 범위가 섞이지 않았는가
- 로컬 신고/차단 상태가 이후 서버 전환을 막는 방향으로 과하게 굳어지지 않았는가

## 원하는 답변 형식

반드시 아래 형식대로 답변 작성해라.

### 1. 핵심 발견사항

- 심각도 순서대로 정리
- 관련 파일 경로를 함께 적기
- 왜 문제가 되는지 짧게 설명

### 2. 신고/차단 최소 골격 평가

- 적절함 / 약간 위험함 / 더 보완 필요
- 이유를 3~5줄로 설명

### 3. 문서 / 코드 충돌 여부

- 충돌 있음 / 없음
- 있다면 어떤 문서와 어떤 코드가 어긋나는지 구체적으로

### 4. 저장 / 필터링 경계 평가

- 신고/차단 로컬 상태, 읽기 필터링, 프로필 차단 목록 경계가 현재 단계에서 적절한지 평가
- 다음 단계 서버 moderation 연결 시 위험할 수 있는 부분이 있으면 지적

### 5. 지금 단계에서 구현하지 말아야 할 것

- 다음 단계로 미뤄야 하는 항목
- 현재 코드에 추가되면 범위를 키우는 항목

### 6. 바로 보완할 것

- 지금 당장 수정 추천 3개 이내

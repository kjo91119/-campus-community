# 20. 작업 완료 후 교차검증 지시서

문서 목적: 구현 작업이 끝난 뒤, 설계 문서와 실제 결과물이 일치하는지 다른 개발자나 다른 모델이 독립적으로 검토할 수 있도록 최종 교차검증 지시문을 제공한다.

## 가정 (Assumptions)

- 이 문서는 기능 구현이 한 차례 완료된 뒤 사용한다.
- 검토자는 구현 담당자와 분리된 시선으로 리뷰한다.
- 검토의 우선순위는 "멋진 구현"보다 "설계 대비 정합성"이다.

## 이 지시서를 사용하는 시점

- MVP 1차 구현 완료 직후
- 베타 배포 전
- 큰 기능 묶음이 하나 끝난 시점
- Supabase 연동 후 회귀 검토 시점

## 검토자가 확인해야 하는 핵심 원칙

1. 이 앱은 대학생 전체 커뮤니티가 아니라 보건의료기사 계열 전공 네트워크여야 한다.
2. 학교는 핵심 체류 가치가 아니라 인증과 제한형 학교 보드의 장치여야 한다.
3. 4개 전공군 동시 런칭 구조에서도 통합 홈과 통합 모집이 중심이어야 한다.
4. MVP에서는 과도한 확장 기능보다 핵심 루프 완성 여부를 우선 본다.
5. 설계 문서에 없는 기능이 많아졌다면 "좋아 보이더라도" 범위 이탈로 본다.

## 검토 목표

### 1. 정체성 검증

- 홈과 주요 탐색 구조가 전공 네트워크 중심인가
- 학교 보드가 메인 기능처럼 과도하게 커지지 않았는가
- 물리치료, 작업치료, 방사선, 임상병리 4개 전공군이 통합 구조 안에서 다뤄지는가

### 2. MVP 범위 검증

- [docs/02-mvp-scope.md](./02-mvp-scope.md)의 MVP 필수 항목이 실제로 구현되었는가
- 실시간 채팅, 과도한 소셜 기능, 불필요한 랭킹 등 MVP 제외 항목이 섞이지 않았는가
- 반대로 필수 인증/피드/작성/신고 기능이 빠지지 않았는가

### 3. 사용자 플로우 검증

다음 플로우를 실제로 따라가며 확인한다.

- 가입/인증
- 온보딩
- 통합 홈 진입
- 전공 필터 탐색
- 학교 보드 진입
- 글쓰기
- 댓글/대댓글
- 모집글 작성 및 참여
- 신고/차단

각 플로우에서 아래를 본다.

- 중간에 막히는 단계가 없는가
- 권한 부족 상태와 빈 상태 처리가 있는가
- 문서와 다르게 불필요한 단계가 추가되지 않았는가

### 4. IA 및 화면 검증

- 바텀탭과 상위 화면 구조가 [docs/05-information-architecture.md](./05-information-architecture.md)와 크게 어긋나지 않는가
- 홈, 모집, 학교, 프로필의 역할이 뒤섞이지 않는가
- 첫 진입 시 앱이 비어 보이지 않는가
- [docs/09-screen-list-and-prd.md](./09-screen-list-and-prd.md)의 우선 화면이 실제 구현되었는가

### 5. 데이터 및 권한 검증

- 실제 스키마 또는 타입 구조가 [docs/07-supabase-data-model-draft.md](./07-supabase-data-model-draft.md)의 핵심 엔터티를 크게 벗어나지 않는가
- `profiles`, `verifications`, `boards`, `posts`, `comments`, `reports`, `blocks`, `recruitments`가 최소 수준으로 준비되었는가
- 학교 보드 접근이 인증/권한 기준으로 제어되는가
- 통합 홈과 전공 필터를 지원하기 어려운 데이터 구조가 되지 않았는가

### 6. 안전 및 운영 검증

- 학교 이메일 인증 또는 수동 인증 보조 플로우가 존재하는가
- 공개 노출이 닉네임 중심인가
- 신고/차단이 동작하는가
- 민감정보 노출 위험이 없는가
- 운영 검토가 필요한 상태값이 정의되어 있는가

### 7. 분석 검증

- 가입, 인증, 온보딩 완료, 첫 글, 첫 댓글, 모집 참여를 추적할 기반이 있는가
- D1/D7 리텐션을 계산할 최소 이벤트 구조가 있는가
- 운영상 필요한 로그와 상태값이 남는가

## 교차검증 체크리스트

### 제품과 범위

- [ ] 전공 네트워크 정체성이 명확하다
- [ ] 학교는 제한형 보조 축으로 유지된다
- [ ] 통합 홈이 실제로 작동한다
- [ ] 통합 모집 또는 이에 준하는 구조가 있다
- [ ] MVP 제외 기능이 과도하게 추가되지 않았다

### 사용자 플로우

- [ ] 인증 완료까지 막힘이 없다
- [ ] 온보딩에서 닉네임과 전공군 설정이 가능하다
- [ ] 홈에서 전공 필터가 작동한다
- [ ] 글 상세와 댓글 흐름이 닫혀 있다
- [ ] 학교 보드 접근 제어가 맞다
- [ ] 모집글 작성과 참여가 가능하다
- [ ] 신고/차단이 노출되고 동작한다

### 데이터와 권한

- [ ] 보드 스코프가 통합/전공군/학교 기준으로 구분된다
- [ ] 인증 상태에 따라 읽기/쓰기 권한이 나뉜다
- [ ] 학교 보드 접근 제어가 서버 또는 이에 준하는 안전한 방식으로 보장된다
- [ ] 차단 사용자 콘텐츠 처리 기준이 존재한다

### 운영과 분석

- [ ] 수동 인증 검토 상태가 구분된다
- [ ] 민감정보 노출이 최소화된다
- [ ] 핵심 분석 이벤트를 추적할 수 있다
- [ ] 출시 전 QA에 필요한 상태 구분이 있다

## 검토 결과 작성 방식

검토 결과는 반드시 발견사항 중심으로 작성한다.

### 작성 원칙

1. 문제를 심각도 순으로 나열한다.
2. 근거가 되는 파일 경로, 화면, 동작을 함께 적는다.
3. 어떤 설계 문서와 충돌하는지 같이 적는다.
4. 단순 감상보다 리스크와 영향도를 설명한다.
5. 큰 문제가 없으면 "치명적 발견사항 없음"을 명시한다.

### 권장 출력 템플릿

```md
# 교차검증 결과

## Findings

1. [심각도: High] 홈 구조가 전공 네트워크보다 학교 보드 중심으로 치우쳐 있음
- 근거: [실제 파일/화면]
- 충돌 문서: [docs/01-product-definition.md], [docs/05-information-architecture.md]
- 이유: 학교는 보조 축이어야 하나 현재는 메인 체류 공간처럼 보여 제품 정체성이 흐려짐
- 영향: IA 재작업 가능성, 초기 포지셔닝 혼선

2. [심각도: Medium] 모집 참여 후 상태 피드백이 없음
- 근거: [실제 파일/화면]
- 충돌 문서: [docs/04-user-flows.md], [docs/09-screen-list-and-prd.md]
- 영향: 사용자 혼란, 참여 전환 저하

## Open Questions

- [남은 의사결정 항목]

## Summary

- MVP 핵심 루프는 대체로 구현됨 / 미완성
- 출시 전 반드시 수정할 항목: ...
```

## 최종 판정 기준

아래 조건을 대부분 만족하면 "설계 대비 구현 정합성 양호"로 본다.

- 전공 네트워크 정체성이 분명하다
- MVP 필수 기능이 빠지지 않았다
- MVP 제외 기능이 과도하게 들어오지 않았다
- 통합 홈과 전공 또는 필터 구조가 실제로 작동한다
- 학교 보드는 제한형 역할로 유지된다
- 인증/신고/차단의 최소 안전장치가 존재한다
- 핵심 지표를 추적할 최소 기반이 있다

## 결론

교차검증은 단순 QA가 아니라 "처음 합의한 방향에서 벗어나지 않았는가"를 보는 절차다. 이 프로젝트에서는 특히 전공 네트워크 중심성, 통합 홈 구조, MVP 범위 유지, 인증 기반 신뢰 설계 네 가지를 가장 우선 기준으로 삼아야 한다.

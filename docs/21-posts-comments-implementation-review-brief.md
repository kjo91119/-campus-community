# 21. 게시글/댓글 구현 교차검증 지시서

문서 목적: 다른 AI 또는 리뷰어가 현재 `campus-community` 저장소에 추가된 게시글 작성 및 댓글 구현을 코드와 문서 기준으로 교차검증할 때, 무엇을 어떤 기준으로 봐야 하는지 명확하게 전달한다.

아래 내용을 그대로 복사해서 다른 모델에게 전달해도 된다.
단, 이 지시서만 보내면 충분한 것이 아니라 검토 대상 코드와 관련 문서 원문도 함께 제공하거나, 같은 저장소/워크스페이스에 접근 가능한 환경에서 사용해야 한다.

---

너는 지금 Expo + TypeScript 기반 앱 프로젝트 `campus-community`의 외부 코드 리뷰어다.

이번 검토의 목적은 "새 기능 제안"이 아니라, 방금 구현된 게시글 작성 및 댓글 단계가 다음 기준을 만족하는지 점검하는 것이다.

## 현재 구현 목표 요약

이번 단계에서 구현한 것은 아래 범위다.

- 게시글 작성 기능
- 댓글 목록 및 댓글 작성 기능
- 전공 게시판과 학교 게시판에 글을 올릴 수 있는 기본 구조
- 로그인 필요, 온보딩 완료 필요, 읽기 전용 제한, 학교 보드 제한을 포함한 권한 분기
- MVP 기준 최소 validation

중요:

- 아직 게시글/댓글의 Supabase 실연결은 하지 않았다.
- 이번 단계는 `AsyncStorage` 기반 로컬 저장 계층과 라우팅 흐름을 먼저 구현했다.
- 대댓글 신규 작성, 좋아요, 첨부파일, 게시글 수정/삭제는 이번 단계 범위가 아니다.
- UI 화려함보다 저장 안정성, 권한 분기, 화면 흐름, 다음 단계 Supabase 연결성을 우선 검토해야 한다.

## 리뷰 목표

아래 항목을 중심으로 검토해라.

1. 게시글 작성과 댓글 작성 흐름이 현재 앱 구조에서 안정적으로 동작하는가
2. 통합 홈, 전공 게시판, 학교 게시판 간 글쓰기 진입 구조가 문서 기준과 맞는가
3. 권한 분기 로직이 `로그인 / 인증 완료 / 온보딩 완료 / 읽기 전용 / 학교 일치` 조건을 일관되게 반영하는가
4. `use-community-data.tsx`의 로컬 저장 구조가 다음 단계 Supabase 연결 전까지 충분히 안정적인가
5. 댓글/게시글 validation이 MVP 최소 기준으로 적절한가
6. 이번 단계에 범위를 키우는 기능이 과하게 섞이지 않았는가
7. 다음 단계에서 `posts` / `comments` Supabase 연결로 자연스럽게 넘어갈 수 있는가

## 특히 강하게 봐야 할 포인트

### A. 라우팅과 진입 구조

- 홈 탭에서 통합 홈 글쓰기, 전공 게시판 진입, 학교 게시판 진입이 자연스럽게 이어지는가
- `app/(tabs)/boards/[boardId].tsx`, `app/(tabs)/posts/[postId].tsx`, `app/(tabs)/write.tsx`가 redirect loop 없이 동작하는가
- 게시판이 없거나 게시글이 없을 때 fallback과 빈 상태가 적절한가

### B. 권한 분기

- `use-community-data.tsx`가 `canAccessCommunity`, `isAuthenticated`, `isReadOnly`, `primaryUniversityId`를 이용해 권한을 제대로 제어하는가
- 학교 게시판 글쓰기가 같은 학교 사용자에게만 열리는가
- 읽기 전용 사용자는 읽기만 가능하고 쓰기는 막히는가
- UI에서 막는 것뿐 아니라 저장 함수에서도 한 번 더 검증하는가

### C. 저장 경계

- `AsyncStorage` 기반 게시글/댓글 저장이 현재 단계에 적절한가
- hydrate 시 캐시 손상이나 기본 mock 데이터와의 충돌 위험이 큰가
- 지금 구조가 Supabase 실연결 때 갈아엎어야 할 정도로 임시성이 큰가, 아니면 repository 교체 수준으로 이어질 수 있는가

### D. 타입과 데이터 구조

- `CommunityPost`, `Comment`, `Board`를 현재 단계에서 무리 없이 재사용하고 있는가
- `posts`와 `comments`가 문서의 데이터 모델 초안과 크게 어긋나지 않는가
- 댓글 `kind`, 게시판 `scopeType`, 학교/전공군 스냅샷 필드가 이후 권한/조회 구조에 맞게 잡혀 있는가

### E. 범위 관리

- 대댓글 신규 작성, 좋아요, 첨부파일, 수정/삭제 등 이번 단계에서 제외한 기능이 섞이지 않았는가
- 모집글 별도 신청 시스템 같은 과한 구조가 들어오지 않았는가
- 학교 게시판이 메인 체류 공간처럼 과도하게 커지지 않았는가

## 원하는 응답 형식

반드시 아래 형식으로 답해라.

### 1. 핵심 발견사항

- 심각도 순서대로 정리
- 각 항목마다 관련 파일명을 명시
- 왜 문제가 되는지 짧고 명확하게 설명

### 2. 게시글/댓글 구조 평가

- 적절함 / 약간 위험함 / 더 보완 필요
- 이유를 3~5줄로 설명

### 3. 문서 / 코드 / 라우팅 충돌 여부

- 충돌 있음 / 없음
- 있다면 어떤 문서와 어떤 코드가 어긋나는지 구체적으로

### 4. 저장 경계 평가

- 로컬 저장 구조가 현재 단계에서 적절한지 평가
- 다음 단계 Supabase 연결 시 위험할 수 있는 부분이 있으면 지적

### 5. 지금 단계에서 구현하지 말아야 할 것

- 게시글/댓글 단계 이후로 미뤄야 하는 항목
- 현재 코드에 추가되면 범위를 키우는 항목

### 6. 바로 보완할 것

- 지금 당장 수정 추천 3개 이내

### 7. 최종 판단

아래 두 문장을 반드시 포함해라.

1. "이 구현 상태로 게시글/댓글 Supabase 연결 단계에 들어가도 되는가?"
2. "들어가기 전에 먼저 수정해야 할 상위 3개 항목"

## 리뷰 원칙

- 칭찬보다 권한 누수, 라우팅 문제, 저장 경계, 문서-코드 불일치, 범위 이탈 가능성을 우선 지적해라.
- 스타일 취향보다 글쓰기 흐름 완결성, 댓글 일관성, 학교 보드 제한, 다음 단계 연결성을 더 중요하게 봐라.
- 실제 코드 리뷰처럼 validation 누락, 상태 동기화, 잘못된 보드 접근, 캐시 손상 복구, 과범위 확장 가능성을 중심으로 보라.

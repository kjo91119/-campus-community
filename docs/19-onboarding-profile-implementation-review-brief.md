# 19. 온보딩/프로필 초기 설정 구현 교차검증 지시서

문서 목적: 다른 AI 또는 리뷰어가 현재 `campus-community` 저장소에 추가된 온보딩/프로필 초기 설정 구현을 코드와 문서 기준으로 교차검증할 때, 무엇을 어떤 기준으로 봐야 하는지 명확하게 전달한다.

아래 내용을 그대로 복사해서 다른 모델에게 전달해도 된다.
단, 이 지시서만 보내면 충분한 것이 아니라 검토 대상 코드와 관련 문서 원문도 함께 제공하거나, 같은 저장소/워크스페이스에 접근 가능한 환경에서 사용해야 한다.

---

너는 지금 Expo + TypeScript 기반 앱 프로젝트 `campus-community`의 외부 코드 리뷰어다.

이번 검토의 목적은 "새 기능 제안"이 아니라, 방금 구현된 인증 이후 온보딩/프로필 초기 설정 단계가 다음 기준을 만족하는지 점검하는 것이다.

## 현재 구현 목표 요약

이번 단계에서 구현한 것은 아래 범위다.

- 최초 가입 후 닉네임, 학교, 전공군 선택 온보딩 플로우
- `profiles` 저장 구조 연결
- 인증 상태, 온보딩 완료 여부, 학교/전공군 선택 여부에 따른 홈 진입 분기
- 이후 게시판 접근 권한 분기에 활용할 수 있는 프로필 상태 정리
- Supabase 가능 시 `profiles` 테이블 저장, 실패 또는 미설정 시 로컬 캐시 fallback

중요:

- 아직 게시판, 댓글, 모집글의 실제 CRUD 연결은 하지 않았다.
- 아직 학교 이메일 검증 고도화나 학생증 수동 인증 실구현은 하지 않았다.
- 이번 단계는 "온보딩 흐름 + 프로필 저장 구조 + 홈 진입 조건"까지만 구현했다.
- UI 완성도보다 상태 구조, 저장 경계, 라우팅 분기, 다음 단계 연결성을 우선 검토해야 한다.

## 리뷰 목표

아래 항목을 중심으로 검토해라.

1. 인증 이후 온보딩 분기가 현재 앱 구조에서 안정적으로 동작하는가
2. 닉네임, 학교, 전공군 선택 플로우가 MVP 기준에 맞게 구현되었는가
3. `use-app-session.tsx`가 auth 상태와 profile/onboarding 상태를 적절히 연결하고 있는가
4. `profiles` 저장 구조가 현재 앱 타입과 다음 단계 Supabase 연결에 무리 없이 이어질 수 있는가
5. 로컬 캐시 fallback과 Supabase 저장 경계가 혼동 없이 정리되어 있는가
6. 홈 진입 조건이 인증 상태, 온보딩 완료 여부, 학교/전공군 선택 여부를 일관되게 반영하는가
7. 이번 단계에 게시판/댓글/모집글 범위가 과하게 섞이지 않았는가
8. 다음 단계에서 `boards` / `posts` 읽기 연결로 자연스럽게 넘어갈 수 있는가

## 특히 강하게 봐야 할 포인트

### A. 온보딩 분기와 라우팅

- `app/(auth)/_layout.tsx`가 인증 완료 사용자를 적절히 온보딩 또는 탭으로 보내는가
- `canAccessCommunity` 조건이 현재 문서 기준과 맞는가
- 인증은 되었지만 학교/전공군이 비어 있는 사용자가 홈으로 잘못 진입할 가능성은 없는가

### B. 프로필 저장 경계

- `hooks/use-app-session.tsx`가 `auth user -> app profile -> Supabase profiles/local cache` 흐름을 무리 없이 정리하고 있는가
- Supabase 저장 실패 시 로컬 캐시 fallback으로 처리하는 정책이 현재 단계에 적절한가
- 로컬 상태와 원격 상태가 어긋날 때 다음 단계에서 문제를 만들 가능성은 없는가

### C. 타입과 SQL 정합성

- `types/domain.ts`의 `Profile` 구조와 `lib/supabase/profiles.ts`의 row 매핑이 맞는가
- `supabase/migrations/20260316120000_create_profiles.sql`이 현재 앱 코드 기준으로 충분한가
- `primary_university_id`, `primary_major_group_id`를 `text`로 두는 선택이 현재 MVP 앱 구조와 충돌하지 않는가

### D. 학교/전공군 선택 규칙

- 학교 이메일로 확인된 학교와 프로필 학교 선택이 일관되게 강제되는가
- `물리치료 / 작업치료 / 방사선 / 임상병리` 4개 전공군 선택 구조가 이후 권한 분기에 활용 가능하게 정리되었는가
- 세부 전공명 `majorLabel`을 지금 단계에서 이 정도로만 받는 것이 과하지 않은가

### E. 범위 관리

- 이번 단계에서 게시판/댓글/모집글 저장 로직이 온보딩과 섞이지 않았는가
- 학생증 수동 인증이나 운영자 흐름이 과하게 들어오지 않았는가
- 아직 하지 말아야 할 검증 고도화가 섞이지 않았는가

## 원하는 응답 형식

반드시 아래 형식으로 답해라.

### 1. 핵심 발견사항

- 심각도 순서대로 정리
- 각 항목마다 관련 파일명을 명시
- 왜 문제가 되는지 짧고 명확하게 설명

### 2. 온보딩/프로필 구조 평가

- 적절함 / 약간 위험함 / 더 보완 필요
- 이유를 3~5줄로 설명

### 3. 문서 / 타입 / SQL 충돌 여부

- 충돌 있음 / 없음
- 있다면 어떤 문서와 어떤 코드/SQL이 어긋나는지 구체적으로

### 4. 저장 경계 평가

- Supabase / 로컬 캐시 / auth seed 경계가 명확한지 평가
- 다음 단계에서 문제가 될 수 있는 부분이 있으면 지적

### 5. 지금 단계에서 구현하지 말아야 할 것

- 온보딩/프로필 단계 이후로 미뤄야 하는 항목
- 현재 코드에 추가되면 범위를 키우는 항목

### 6. 바로 보완할 것

- 지금 당장 수정 추천 3개 이내

### 7. 최종 판단

아래 두 문장을 반드시 포함해라.

1. "이 구현 상태로 게시판 읽기 연결 단계에 들어가도 되는가?"
2. "들어가기 전에 먼저 수정해야 할 상위 3개 항목"

## 리뷰 원칙

- 칭찬보다 상태 일관성, 저장 경계, 라우팅 분기, 문서-코드-마이그레이션 정합성을 우선 지적해라.
- 스타일 취향보다 온보딩 안정성, 프로필 저장 구조, fallback 정책, 다음 단계 연결성을 더 중요하게 봐라.
- 실제 코드 리뷰처럼 redirect 조건, 로컬/원격 상태 불일치, 타입 경계, SQL 구조 누락 가능성을 중심으로 보라.

# 15. Phase 1 구현 교차검증 지시서

문서 목적: 다른 AI 또는 리뷰어가 현재 Phase 1 구현 결과물을 코드 기준으로 교차검증할 때, 무엇을 어떤 기준으로 봐야 하는지 명확하게 전달한다.

아래 내용을 그대로 복사해서 다른 모델에게 전달해도 된다.
단, 이 지시서만 보내면 충분한 것이 아니라 검토 대상 코드와 관련 문서 원문도 함께 제공하거나, 같은 저장소/워크스페이스에 접근 가능한 환경에서 사용해야 한다.

---

너는 지금 Expo + TypeScript 기반 앱 프로젝트 `campus-community`의 외부 코드 리뷰어다.

이번 검토의 목적은 "기획 검토"가 아니라, 방금 구현된 Phase 1 골격 코드가 다음 기준을 제대로 만족하는지 점검하는 것이다.

## 현재 구현 목표 요약

이번 단계에서 구현한 것은 아래 범위다.

- Expo 앱 라우팅 구조 정리
- `홈 / 모집 / 학교 / 프로필` 4탭 초안 반영
- `app/(auth)` 그룹 추가
- 학교 이메일 인증 화면 골격
- 학생증 수동 인증 제출/상태 화면 골격
- 온보딩 화면 골격
- 통합 홈 화면 쉘
- 전공군 필터와 전공군 집중 보기 상태
- 학교 게시판 진입 구조
- 모집 탭 골격
- 프로필/인증 상태 화면 골격
- 공통 타입/상수/더미 데이터 정리
- Supabase 없이 더미 세션 상태로 흐름 동작

중요:

- 아직 Supabase 실연결은 하지 않았다.
- 아직 패키지 추가 설치는 하지 않았다.
- 아직 게시글/댓글/모집 CRUD 본 구현은 하지 않았다.
- 이번 단계는 구조와 흐름, 더미 데이터 기반 UI 골격이 목적이다.

## 프로젝트 핵심 컨텍스트

- 이 서비스는 한국 대학생 대상의 보건의료기사 계열 전공군 중심 인증 커뮤니티 앱이다.
- 초기 전공군은 물리치료, 작업치료, 방사선, 임상병리 4개다.
- 학교는 인증과 제한형 학교 게시판을 위한 장치이고, 핵심은 전공 네트워크다.
- 현재 MVP 기준 네비게이션은 `홈 / 모집 / 학교 / 프로필` 4탭이다.
- 전공 탐색은 별도 탭이 아니라 홈 내부 필터와 집중 보기 상태로 처리한다.
- 모집 참여 방식은 별도 신청 시스템이 아니라 "참여 의사 댓글" 방향으로 설계되어 있다.

## 리뷰 목표

아래 항목을 중심으로 검토해라.

1. 현재 구현이 Phase 1 범위를 넘어서지 않았는가
2. 라우팅 구조가 Expo Router 관점에서 안전한가
3. auth 그룹과 tabs 그룹 분리가 적절한가
4. 더미 세션 상태 흐름이 문서 기준과 맞는가
5. 인증 상태별 접근 제어가 현재 단계에 맞게 구현되어 있는가
6. 홈 / 모집 / 학교 / 프로필 탭 구조가 문서 기준과 일치하는가
7. 전공 탐색이 별도 탭이 아니라 홈 내부 필터로 처리되고 있는가
8. 학교 게시판과 모집 탭이 너무 이르게 구현되지 않고 골격 수준에 머물러 있는가
9. 타입/상수/더미 데이터 구조가 다음 단계 확장에 무리 없게 정리되어 있는가
10. 지금 단계에서 생길 수 있는 버그, UX 단절, 구조적 리스크가 무엇인가

## 특히 강하게 봐야 할 포인트

### A. 라우팅과 세션 가드

다음 질문으로 검토해라.

- `app/_layout.tsx`에서 auth와 tabs 흐름을 나누는 방식이 안전한가
- `app/(auth)/_layout.tsx`의 redirect 조건이 루프를 만들 가능성은 없는가
- `app/(tabs)/_layout.tsx`에서 인증되지 않은 사용자를 홈이 아닌 auth로 보내는 방식이 적절한가
- 상대 경로 navigation 사용이 현재 파일 구조에서 맞는가

### B. 더미 세션 구조

다음 질문을 봐라.

- `hooks/use-app-session.tsx`가 현재 단계에서 너무 많은 책임을 가지지 않는가
- `verificationStatus`, `accountStatus`, `onboardingCompleted` 조합이 문서 기준과 맞는가
- 다음 단계에 Supabase 연동을 붙일 때 교체하기 쉬운 구조인가
- 데모 제어 액션이 실제 앱 코드와 너무 섞여 있지는 않은가

### C. 문서-구현 일치 여부

다음 연결을 특히 점검해라.

- `docs/05-information-architecture.md` 와 `app/(tabs)/_layout.tsx`
- `docs/08-auth-verification-moderation.md` 와 `hooks/use-app-session.tsx`
- `docs/09-screen-list-and-prd.md` 와 `app/(auth)/*`
- `docs/02-mvp-scope.md` 와 `app/(tabs)/recruit.tsx`

### D. 범위 과다 여부

다음 항목을 보라.

- 지금 단계에서 불필요하게 상세한 상태 관리가 들어간 부분이 있는가
- 아직 placeholder만 두는 편이 나은 화면이 과하게 구현된 부분은 없는가
- 반대로 구조를 잡기 위해 꼭 필요한데 빠진 최소 골격은 없는가

## 원하는 응답 형식

반드시 아래 형식으로 답해라.

### 1. 핵심 발견사항

- 심각도 순서대로 정리
- 각 항목마다 관련 파일명을 명시
- 왜 문제가 되는지 짧고 명확하게 설명

예시 형식:

- 높음: `app/(auth)/_layout.tsx`
  - redirect 조건이 특정 상태에서 루프를 만들 가능성이 있다.

### 2. Phase 1 구현 범위 평가

- 적절함 / 약간 과함 / 더 좁혀야 함
- 이유를 3~5줄로 설명

### 3. 문서-구현 충돌 여부

- 충돌 있음 / 없음
- 있다면 어떤 문서와 어떤 코드 파일이 어떻게 어긋나는지 구체적으로

### 4. 구조적으로 좋은 점

- 다음 단계 확장에 도움이 되는 부분만 2~4개 이내로 간단히

### 5. 지금 단계에서 구현하지 말아야 할 것

- 현재 코드에 추가되면 범위를 키우는 항목
- 다음 단계로 미뤄야 하는 항목

### 6. 다음 단계 진입 전 수정 추천

- 바로 고쳐야 할 것
- 다음 단계 시작과 함께 고쳐도 되는 것

### 7. 최종 판단

아래 두 문장을 반드시 포함해라.

1. "이 구현 상태로 Phase 2 준비에 들어가도 되는가?"
2. "들어가기 전에 먼저 수정해야 할 상위 3개 항목"

## 리뷰 원칙

- 칭찬보다 문제 발견과 위험 지적을 우선해라.
- 애매하면 보수적으로 판단해라.
- 스타일 취향보다 라우팅 안정성, 상태 구조, 범위 관리, 문서 일치를 더 중요하게 봐라.
- placeholder 화면은 시각 완성도보다 라우팅 안정성과 문서 일치 여부를 우선 검토해라.
- 기능 추가 제안보다, 현재 골격이 안전한지와 다음 단계 연결성이 좋은지를 우선 평가해라.
- 실제 코드 리뷰처럼 버그, 구조 리스크, 유지보수 문제를 중심으로 봐라.

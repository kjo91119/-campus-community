# 12. 오픈 퀘스천

문서 목적: 아직 결정되지 않은 제품, 운영, 기술 이슈를 정리하고, 각 항목마다 현재 시점의 추천 방향을 함께 제시한다.

## 가정 (Assumptions)

- 아래 항목은 모두 구현 전에 결정될수록 좋지만, 일부는 베타 테스트를 통해 최종 확정할 수 있다.

## 이번 라운드에서 고정한 결정

- MVP 바텀탭은 `홈 / 모집 / 학교 / 프로필` 4개로 고정한다.
- 전공군 탐색은 홈 내부 필터와 전공군 집중 보기 상태로 처리하고, 별도 전공 탭은 V1 이후 재검토한다.
- 모집 참여 방식은 별도 신청 테이블 없이 "참여 의사 댓글"로 통일한다.
- 수동 인증 증빙은 최종 결정 후 7일 내 삭제하고, 분쟁이 열릴 때만 최대 30일까지 보관한다.
- 수동 인증은 영업일 기준 24시간 내 접수 확인, 72시간 내 최종 승인/반려를 목표로 한다.
- 신고는 긴급 6시간, 일반 24시간 1차 확인 / 72시간 내 처리를 기준으로 한다.

## 제품 측면

### 1. 학교 보드의 비중을 어디까지 둘 것인가

선택지:

- A. 학교별 단일 보드만 제공
- B. 학교별 보드를 세분화

추천 방향:

- A를 추천한다.

이유:

- MVP에서는 학교 보드가 너무 커지면 전공 네트워크 정체성이 약해진다.
- 학교별 세분화는 빈 게시판 문제를 악화시킨다.

### 2. V1 이후 전공 탭을 별도로 둘 필요가 있는가

선택지:

- A. 홈 내부 필터만 사용
- B. 전공 탭을 별도로 둔다

현재 결정:

- MVP는 A로 고정한다.

향후 추천 방향:

- 홈 내부 전공군 필터 사용률이 높고 콘텐츠 밀도가 충분하면 B를 검토한다.

이유:

- 현재 MVP에서는 홈 중심 구조가 더 단순하고 구현 리스크가 낮다.
- 전공 탭을 너무 일찍 분리하면 빈 화면과 네비게이션 복잡도만 늘어날 수 있다.

### 3. 학교명 공개 범위를 어떻게 할 것인가

선택지:

- A. 기본 공개
- B. 기본 비공개, 인증만 보유
- C. 일부 화면에서만 공개

추천 방향:

- B 또는 C 중 보수적으로 시작한다.

이유:

- 학교는 신뢰 장치이지 메인 노출 정보가 아니다.

## 운영 측면

### 4. 운영 도구를 언제 별도 관리자 도구로 분리할 것인가

선택지:

- A. Supabase + 운영 체크리스트 문서로 계속 운영
- B. 리뷰 큐가 늘어나면 별도 관리자 도구를 만든다

추천 방향:

- A로 시작하고, 수동 인증과 신고가 누적될 때 B를 검토한다.

이유:

- MVP 시점에서는 도구 구축보다 정책과 SLA 고정이 더 중요하다.

### 5. 모집 허위 글 대응을 어떻게 시작할 것인가

선택지:

- A. 신고 기반 수동 대응
- B. 별도 사전 검수

추천 방향:

- A를 추천한다.

이유:

- MVP에서 사전 검수는 운영 비용이 크다.

### 6. 학교 이메일 미지원 대학을 어떻게 처리할 것인가

선택지:

- A. 수동 인증만 제공
- B. 출시 초기에는 제외

추천 방향:

- A를 추천한다.

이유:

- 너무 일찍 제외하면 잠재 사용자층이 줄어든다.

## 기술 측면

### 7. `majors`를 얼마나 세밀하게 관리할 것인가

선택지:

- A. MVP에서는 전공군 중심, 세부 학과명은 자유 입력
- B. 학교별 학과 마스터까지 구축

추천 방향:

- A를 추천한다.

이유:

- 학교별 학과명 매핑은 초기 비용이 크다.

### 8. 알림 테이블을 MVP부터 만들 것인가

선택지:

- A. 스키마만 미리 두고 UI는 나중
- B. 완전히 나중에 추가

추천 방향:

- A를 추천한다.

이유:

- 댓글/답글 이벤트 확장에 대비할 수 있다.

### 9. 학교 보드 권한을 클라이언트에서만 막을 것인가

선택지:

- A. 클라이언트 제어 중심
- B. RLS로 서버에서 강제

추천 방향:

- B를 추천한다.

이유:

- 권한 로직은 최종적으로 서버에서 보장해야 한다.

### 10. V1에서 모집 참여를 별도 신청 테이블로 분리할 것인가

선택지:

- A. 댓글/간단 지원 의사로 시작
- B. 별도 메시지 시스템 구축

현재 결정:

- MVP는 A로 고정한다.

향후 추천 방향:

- 모집 규모가 커지고 참여/선정 상태를 구조화할 필요가 생기면 B를 검토한다.

이유:

- 댓글 기반 참여는 가장 단순하고 운영 설명도 쉽다.
- 별도 신청 테이블은 상태 관리와 UI 복잡도를 키운다.

## 나중에 반드시 결정해야 할 항목

- 학교명 공개 정책 최종안
- 학교 보드 세분화 여부
- 전공군 확장 기준 수치화
- 운영 도구 분리 시점

## 결론

대부분의 오픈 퀘스천은 "더 많은 기능을 넣을 것인가"보다 "얼마나 좁고 안전하게 시작할 것인가"에 관한 문제다. 현재 시점에서는 보수적으로, 그리고 전공 네트워크 정체성을 훼손하지 않는 방향으로 결정하는 것이 가장 적절하다.

# 02. MVP 범위

문서 목적: 무엇을 이번 MVP에 포함하고 무엇을 제외할지 명확히 고정해, 범위가 커지는 것을 막는다.

## 가정 (Assumptions)

- MVP의 목표는 모든 기능을 담는 것이 아니라, 핵심 커뮤니티 루프를 검증하는 것이다.
- 4개 전공군 동시 런칭을 전제로 하되, 기능 수는 최소화한다.

## MVP 판단 기준

아래 4가지 중 3개 이상을 만족하지 못하면 MVP에서 제외한다.

1. 전공 네트워크 핵심 가치와 직접 연결되는가
2. 초기 사용자가 첫 주 안에 반복 사용할 이유를 만드는가
3. 데이터 구조와 운영 정책이 지나치게 복잡해지지 않는가
4. 4개 전공군 동시 런칭 시 빈 게시판 문제를 악화시키지 않는가

## 범위 구분

### MVP

반드시 있어야 하는 기능이다.

| 영역 | MVP 포함 항목 | 이유 |
| --- | --- | --- |
| 인증 | 학교 이메일 인증, 학생증 수동 인증 보조 플로우 | 신뢰 기반 커뮤니티의 전제 조건 |
| 계정 | 닉네임 설정, 전공군 선택, 기본 프로필(활동 내역 제외) | 공개 노출은 닉네임 중심으로 간다 |
| 홈 | 통합 홈 피드, 전공군 필터, 인기글/최신글 기본 정렬 | 초기 빈 게시판 문제를 줄이는 핵심 |
| 전공 콘텐츠 | 홈 내부 전공군 필터와 전공군 집중 보기 | 별도 전공 탭 없이도 핵심 정체성 구현 가능 |
| 학교 콘텐츠 | 같은 학교 사용자만 보는 학교 보드 1개 | 학교 인증의 실질 가치 제공 |
| 작성/참여 | 글 작성, 댓글, 대댓글 | 커뮤니티 최소 상호작용 루프 |
| 모집 | 모집글 목록, 상세, 작성, 참여 의사 댓글 | 팀모집/스터디 수요가 뚜렷하며 별도 신청 시스템 없이 시작 가능 |
| 안전 | 신고, 차단, 기본 제재 정책 | 초기 운영 리스크 최소화 |
| 운영 | 수동 인증과 신고 처리 SLA, 증빙 보관 기준 | 운영 가능성을 문서 단계에서 먼저 닫아야 함 |

### V1

MVP 이후 빠르게 붙일 수 있는 기능이다.

| 영역 | V1 항목 | 메모 |
| --- | --- | --- |
| 피드 | 저장, 좋아요, 인기 정렬 고도화 | MVP에서는 조회/댓글 중심으로 충분 |
| 알림 | 인앱 알림함, 댓글/답글 알림 | 바텀탭보다는 헤더 진입이 적합 |
| 프로필 | 내가 쓴 글, 저장한 글, 활동 요약 | MVP 이후 재방문 강화용 |
| 모집 | 모집 상태 변경, 신청 관리 UX 개선, 별도 지원 테이블 | MVP는 참여 의사 댓글 기반으로 시작 가능 |
| 운영 | 신고 사유 세분화, 운영 메모 | 운영 규모 증가 시 필요 |

### Later

초기 MVP에서는 의도적으로 제외한다.

| 영역 | Later 항목 | 제외 이유 |
| --- | --- | --- |
| 소셜 | 1:1 채팅, 그룹 채팅, 팔로우 | 범위가 커지고 운영 난이도 증가 |
| 콘텐츠 | 이미지 중심 게시글, 동영상 업로드 | 저장소/검수/정책 복잡도 증가 |
| 게임화 | 뱃지, 레벨, 랭킹 | 핵심 가치 검증 전에 과함 |
| 학교 기능 | 학교별 세부 게시판 다중 운영 | 초기 분산으로 인해 밀도 저하 |
| 확장 | 실무자/졸업생 네트워크, 멘토링 | 커뮤니티 검증 후 고려 |
| 상거래 | 중고거래, 공동구매 | 제품 정체성이 흐려짐 |

## 이번 MVP에서 절대 안 만드는 것

- 실시간 채팅
- 학교별 다중 게시판 세분화
- 전공군별 개별 앱처럼 보이는 별도 구조
- 홈과 분리된 별도 전공 바텀탭
- 지나치게 복잡한 랭킹/포인트 시스템
- 익명 채팅, 오픈채팅 대체 기능
- 광고, 유료 구독, 결제
- 관리자 웹 전체 구축

## MVP의 최소 사용자 가치

사용자는 다음 세 가지를 할 수 있어야 한다.

1. 학교 인증을 거쳐 신뢰 가능한 사용자임을 증명한다.
2. 전공군 통합 홈과 전공 필터를 통해 필요한 정보를 빠르게 찾는다.
3. 글, 댓글, 모집글을 통해 실제 상호작용을 시작한다.

이 세 가지가 되지 않는 기능은 MVP 우선순위에서 밀린다.

## 범위가 커지는 것을 막는 운영 기준

### 콘텐츠 범위

- 글 유형은 일반글, 질문글, 모집글 정도로 시작한다.
- 게시판 카테고리는 실습, 국시, 취업, 질문답변, 자유 정도의 최소 세트만 둔다.
- 학교 보드는 학교별 1개 제한형 보드로 시작한다.
- 모집 참여는 별도 신청 시스템이 아니라 댓글 기반 참여 의사 표현으로 시작한다.

### 사용자 범위

- 초기 대상은 4개 전공군 재학생 중심이다.
- 졸업생, 현업 종사자, 타 전공군은 후순위다.

### 화면 범위

- 홈, 글상세, 글쓰기, 모집, 학교 보드, 프로필 정도로 묶는다.
- 알림, 저장, 상세 검색, 정교한 추천 시스템은 뒤로 미룬다.

## 4개 전공군 동시 런칭 기준 최소 기능 세트

4개 전공군을 동시에 시작하더라도 아래 세트만 있으면 MVP 검증이 가능하다.

- 인증
- 온보딩
- 통합 홈
- 전공군 필터
- 학교 보드
- 글/댓글
- 모집
- 신고/차단

핵심은 "전공군을 늘리면서 기능도 같이 늘리는 것"이 아니라, 전공군은 4개를 열되 기능은 최소로 고정하는 것이다.

## 결론

이 MVP는 "전공 네트워크의 핵심 루프가 작동하는가"를 검증하는 데 집중한다. 범위를 조금만 넓혀도 개발량보다 운영량이 먼저 폭발할 가능성이 높으므로, 초기에는 인증, 피드, 글/댓글, 모집, 신고/차단만으로 충분하다.

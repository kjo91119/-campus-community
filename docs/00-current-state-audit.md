# 00. 현재 상태 점검

문서 목적: 현재 저장소가 어느 수준까지 준비되어 있는지 빠르게 파악하고, 왜 지금 구현보다 설계 문서화가 먼저 필요한지 정리한다.

## 가정 (Assumptions)

- 현재 앱 코드는 제품 방향을 확정한 결과물이 아니라 Expo 스타터를 프로젝트 시작용으로 정리한 수준이다.
- 앞으로 실제 기능 구현은 이 문서 세트의 범위와 우선순위를 기준으로 진행한다.

## 저장소 상태 요약

- 프레임워크: Expo + React Native + TypeScript
- 라우팅: Expo Router
- 현재 앱 구조: 탭 기반 기본 화면과 모달 화면이 있는 스타터 앱
- 검증 상태: `lint`, `typecheck`, `expo-doctor` 기준 정상
- 백엔드 연결: 없음
- 인증/데이터/운영 기능: 없음

## 현재 코드베이스에서 확인된 것

### 이미 있는 것

- Expo Router 기반 앱 골격
- 기본 테마 컴포넌트와 탭 네비게이션
- 최소한의 프로젝트 실행 스크립트
- 문서화 및 본격 설계를 시작할 수 있는 안정적인 개발 시작점

### 아직 없는 것

- 실제 서비스 요구사항이 반영된 화면 설계
- 인증 흐름과 사용자 상태 정의
- 게시판, 모집, 학교 보드에 대한 데이터 구조
- 신고, 차단, 운영 정책
- 분석 이벤트 설계
- Supabase 기준 데이터 모델과 권한 전략

## Expo 기본 구조 요약

- `app/`
  - Expo Router 기반 진입 구조
  - 현재는 기본 탭 화면과 모달 화면만 존재
- `components/`
  - 테마 텍스트, 뷰, 탭 버튼 등 재사용 가능한 기본 UI 컴포넌트
- `constants/`
  - 색상, 폰트 등 테마 상수
- `hooks/`
  - 색상 테마 관련 훅

## 지금 단계에서 중요한 판단

현재 저장소는 "개발을 시작할 수 있는 환경"은 준비되어 있지만, "무엇을 어떤 순서로 만들지"는 아직 정리되지 않았다. 이 상태에서 바로 구현에 들어가면 아래 문제가 발생할 가능성이 높다.

- 전공군 중심 앱인데도 학교 중심 구조로 잘못 굳어질 수 있다.
- 게시판과 모집, 학교 보드의 범위가 뒤섞여 MVP가 커질 수 있다.
- 인증, 익명성, 신고 정책을 늦게 정리하면 나중에 데이터 모델을 다시 뜯어야 한다.
- 4개 전공군 동시 런칭을 고려하지 않으면 각 게시판이 텅 빈 구조가 되기 쉽다.

## 왜 설계가 먼저 필요한가

### 1. 이 앱은 범용 커뮤니티가 아니라 니치 네트워크다

일반 대학생 커뮤니티처럼 모든 학교와 모든 학과를 한 번에 포괄하는 모델이 아니다. "보건의료기사 계열 전공군"이라는 좁고 명확한 정체성이 핵심이므로, 홈 구조와 콘텐츠 분류부터 이 정체성을 반영해야 한다.

### 2. 학교는 핵심 가치가 아니라 신뢰 장치다

학교 인증은 중요하지만 사용자가 계속 머무는 이유는 같은 전공군 학생들과 실습, 국시, 취업, 팀모집 정보를 교류하는 데 있다. 따라서 인증 구조와 콘텐츠 구조를 함께 설계해야 한다.

### 3. 초기 빈 게시판 문제를 구조적으로 풀어야 한다

4개 전공군을 동시에 시작하면 겉보기에는 범위가 넓어지지만, 잘못 나누면 각 보드가 모두 비어 보일 수 있다. 따라서 통합 홈, 통합 인기글, 통합 모집, 전공 필터 구조를 먼저 설계해야 한다.

## 현재 단계의 결론

- 구현을 바로 늘리기보다 MVP 정의와 정보구조 설계가 선행되어야 한다.
- 특히 인증, 게시판 범위, 모집 구조, 학교 보드의 역할은 지금 확정해두는 편이 안전하다.
- 다음 단계는 제품 정의, MVP 범위, 플로우, IA, 데이터 모델 문서를 순서대로 고정하는 것이다.

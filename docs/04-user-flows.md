# 04. 사용자 플로우

문서 목적: 가입부터 인증, 탐색, 작성, 모집, 신고까지 MVP 핵심 흐름을 단계별로 정리해 구현 순서를 명확하게 만든다.

## 가정 (Assumptions)

- 사용자는 인증 전에는 소개, 정책, 인증 관련 화면만 접근할 수 있다고 가정한다.
- MVP에서 로그인 방식은 학교 이메일 인증과 학생증 수동 인증 보조 플로우 중심이다.

## 1. 가입 및 인증 플로우

### 기본 플로우: 학교 이메일 인증

1. 사용자가 앱을 설치하고 시작 화면에 진입한다.
2. 서비스 소개와 전공군 중심 가치 제안을 짧게 확인한다.
3. 사용자가 학교 이메일을 입력한다.
4. 시스템이 이메일 도메인을 대학 목록과 대조한다.
5. 일치하면 인증 메일 또는 OTP를 발송한다.
6. 사용자가 링크 또는 OTP로 인증을 완료한다.
7. 시스템이 인증 상태를 `verified`로 저장한다.
8. 사용자는 온보딩으로 이동한다.

### 예외 케이스

- 학교 이메일이 없는 학교 또는 사용자가 이메일 접근이 불가능하다.
- 입력 이메일 도메인이 지원 대학 목록과 일치하지 않는다.
- 인증 메일이 지연되거나 만료된다.
- 이미 다른 계정에서 사용 중인 학교 이메일이다.

### 대체 플로우: 학생증 수동 인증

1. 사용자가 "학교 이메일 인증이 어렵나요?"를 선택한다.
2. 수동 인증 안내와 개인정보 마스킹 가이드를 확인한다.
3. 학생증 이미지와 최소 메타데이터를 제출한다.
4. 시스템은 상태를 `pending`으로 저장한다.
5. 운영자가 검토 후 승인 또는 반려한다.
6. 승인 시 사용자는 온보딩으로 이동한다.

### 수동 인증 예외 케이스

- 사진이 흐리거나 필요한 정보가 가려져 있다.
- 학생증 형식이 학교 기준과 맞지 않는다.
- 사용자가 민감정보를 과도하게 노출했다.
- 검토 지연으로 대기 시간이 길어진다.

## 2. 온보딩 플로우

1. 인증 완료 후 닉네임을 설정한다.
2. 전공군 4개 중 본인의 주 전공군을 선택한다.
3. 필요 시 세부 학과명 또는 관심 태그를 입력한다.
4. 이용 규칙, 익명성 원칙, 신고 정책을 짧게 확인한다.
5. 홈 피드 진입 전 추천 필터를 기본 세팅한다.
6. 통합 홈으로 이동한다.

### 예외 케이스

- 닉네임 중복 또는 정책 위반
- 전공군 미선택
- 인증은 됐지만 온보딩을 중간에 이탈

## 3. 전공군 통합 홈 진입 플로우

1. 사용자가 홈 탭에 진입한다.
2. 기본값으로 "전체 전공군" 통합 피드가 열린다.
3. 상단에서 전공군 필터 칩을 확인한다.
4. 사용자는 전체, 물리치료, 작업치료, 방사선, 임상병리 중 하나를 선택한다.
5. 피드는 인기글, 최신글, 모집글 또는 카테고리 칩으로 좁혀진다.
6. 사용자는 원하는 글 상세로 이동한다.

### 예외 케이스

- 새 사용자의 경우 콘텐츠가 부족할 수 있다.
- 특정 전공군에만 글이 적을 수 있다.
- 차단한 사용자의 글은 피드에서 제외되어야 한다.

## 4. 전공군 집중 보기 플로우

1. 사용자가 홈에서 전공군 필터를 선택한다.
2. 선택한 전공군 기준 피드가 열린다.
3. 상단 카테고리에서 실습, 국시, 취업, 질문답변, 자유, 모집을 고른다.
4. 목록에서 글을 읽고 댓글 또는 작성으로 이어진다.
5. 사용자는 언제든 전체 피드로 되돌아갈 수 있다.

### 예외 케이스

- 카테고리별 콘텐츠 밀도 차이
- 전공군 변경 후 이전 스크롤 위치 처리
- 작성 권한은 인증 완료 사용자에게만 허용

## 5. 학교 전용 게시판 진입 플로우

1. 사용자가 학교 보드 메뉴를 누른다.
2. 시스템이 같은 학교 인증 여부를 확인한다.
3. 인증된 경우 학교 보드 목록 또는 단일 학교 홈으로 진입한다.
4. 사용자는 학교 생활 질문, 공지성 질문, 수업 관련 글을 읽는다.
5. 학교 한정 글을 작성하거나 댓글을 단다.

### 예외 케이스

- 같은 학교 인증이 완료되지 않은 사용자
- 학교 변경 요청이 필요한 경우
- 학교 보드 접근은 가능하지만 글쓰기 제재 상태인 사용자

## 6. 글쓰기 플로우

1. 사용자가 글쓰기 버튼을 누른다.
2. 작성 위치를 선택한다.
   - 통합 전공 네트워크
   - 특정 전공군
   - 학교 보드
   - 모집
3. 글 유형을 선택한다.
   - 일반
   - 질문
   - 모집
4. 제목, 본문, 카테고리를 입력한다.
5. 학교 보드인 경우 익명 노출 정책을 다시 안내한다.
6. 게시 후 상세 화면으로 이동한다.

### 예외 케이스

- 제목/본문 최소 길이 미달
- 금칙어 또는 신고 누적 유저의 작성 제한
- 같은 내용 반복 도배 감지

## 7. 댓글 및 대댓글 플로우

1. 사용자가 글 상세에 진입한다.
2. 댓글 입력창에서 의견을 작성한다.
3. 필요 시 특정 댓글에 답글을 단다.
4. 작성 완료 후 댓글 목록이 갱신된다.
5. 작성자는 자신의 댓글을 삭제할 수 있다.

### 예외 케이스

- 차단한 사용자와 상호작용 제한
- 신고 누적으로 댓글 블라인드 처리
- 삭제된 부모 댓글에 대한 답글 표시 방식

## 8. 신고 플로우

1. 사용자가 글, 댓글, 모집글에서 더보기 메뉴를 연다.
2. 신고 사유를 선택한다.
3. 필요 시 보충 설명을 입력한다.
4. 신고 접수 완료 메시지를 본다.
5. 시스템은 운영 검토 큐에 적재한다.

### 예외 케이스

- 동일 대상 반복 신고
- 허위 신고 누적 사용자
- 이미 블라인드된 대상 재신고

## 9. 차단 플로우

1. 사용자가 특정 작성자의 프로필 또는 더보기 메뉴를 연다.
2. 차단을 선택한다.
3. 차단 이후 해당 사용자의 글/댓글/모집글을 숨긴다.
4. 사용자는 프로필 설정에서 차단 목록을 볼 수 있다.

### 예외 케이스

- 차단 후 기존 댓글 문맥 붕괴
- 운영자/시스템 계정은 차단 대상 예외 처리 필요

## 10. 팀원 모집 글 등록 및 참여 플로우

### 등록

1. 사용자가 모집 탭 또는 글쓰기에서 모집 유형을 선택한다.
2. 모집 목적을 선택한다.
   - 스터디
   - 과제
   - 공모전
   - 프로젝트
3. 모집 인원, 활동 방식, 기간, 선호 전공군을 입력한다.
4. 게시 후 모집 목록과 상세에 노출된다.

### 참여

1. 사용자가 모집글 상세를 읽는다.
2. 사용자가 "참여 의사 댓글 남기기" CTA를 누른다.
3. 댓글 입력창이 열리고, 참여 의사 댓글을 남긴다.
4. 작성자는 댓글 목록에서 참여 희망자를 확인한다.
5. 후속 연결은 MVP에서는 댓글 또는 게시글 본문에 적은 외부 연락 안내를 통해 단순 처리한다.

### 예외 케이스

- 모집 마감 후 참여 요청
- 모집자 잠수 또는 허위 모집
- 학교 보드 한정 모집 여부 정책 미정
- 댓글이 많아질 경우 실제 참여 의사와 일반 질문이 섞일 수 있음

## 구현 관점에서의 플로우 우선순위

1. 가입/인증
2. 온보딩
3. 통합 홈
4. 글 상세와 댓글
5. 글쓰기
6. 학교 보드
7. 모집
8. 신고/차단

이 순서대로 구현하면 서비스 핵심 루프를 가장 안전하게 검증할 수 있다.

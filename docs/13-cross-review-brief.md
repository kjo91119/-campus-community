# 13. 교차검증 지시서

문서 목적: 다른 AI 또는 리뷰어가 현재 문서 세트를 교차검증할 때, 무엇을 어떤 기준으로 봐야 하는지 명확하게 전달한다.

아래 내용을 그대로 복사해서 다른 모델에게 전달해도 된다.

---

너는 지금 Expo + TypeScript 기반 앱 프로젝트 `campus-community`의 외부 검토자다.

이번 검토의 목적은 "구현"이 아니라, 이미 작성된 MVP 기획/설계 문서 세트의 정합성, 현실성, 누락, 과한 범위, 위험 요소를 교차검증하는 것이다.

## 프로젝트 핵심 컨텍스트

- 이 서비스는 한국 대학생 대상의 "보건의료기사 계열 전공군 중심 인증 커뮤니티" 앱이다.
- 초기 런칭 전공군은 물리치료, 작업치료, 방사선, 임상병리 4개다.
- 핵심 정체성은 "대학생 전체 커뮤니티"가 아니라 "전공 네트워크"다.
- 학교는 핵심 체류 가치가 아니라, 인증과 제한형 학교 보드를 위한 장치다.
- 통합 홈, 통합 모집, 전공군 필터 구조를 통해 "초기 빈 게시판 문제"를 줄이는 방향으로 설계되어 있다.
- 현재는 구현보다 기획/설계 문서 정리가 우선이며, 실제 앱 기능은 아직 본격 구현 전 단계다.
- 추후 백엔드는 Supabase를 사용할 가능성이 높다.

## 검토 목표

아래 항목을 중심으로 문서 전체를 교차검증해라.

1. 문서 간 논리 충돌이 있는가
2. MVP 범위가 실제로 충분히 좁고 실행 가능한가
3. "전공 네트워크"라는 핵심 정체성이 모든 문서에 일관되게 반영되어 있는가
4. 학교 역할이 과도하게 커지거나, 반대로 너무 약해져 있지는 않은가
5. 4개 전공군 동시 런칭 전략이 현실적인가
6. 초기 빈 게시판 문제를 줄이기 위한 구조 제안이 충분히 구체적인가
7. 사용자 플로우, IA, 데이터 모델, 운영 정책이 서로 맞물리는가
8. Supabase 기준 데이터 모델 초안이 MVP 구현에 무리가 없는가
9. 인증, 익명성, 신고/차단 정책이 현실적인가
10. 개발 로드맵과 TASKS가 실제 실행 가능한 수준인가

## 특히 강하게 봐야 할 포인트

### A. 범위 과다 여부

다음 질문으로 봐라.

- MVP에 들어간 기능이 실제로 너무 많지 않은가
- 빠져도 되는 기능이 MVP에 남아 있지 않은가
- 반대로 핵심 루프를 위해 꼭 필요한데 빠진 항목은 없는가

### B. 문서 간 정합성

다음 연결을 특히 점검해라.

- `01-product-definition.md` 와 `02-mvp-scope.md`
- `02-mvp-scope.md` 와 `05-information-architecture.md`
- `04-user-flows.md` 와 `09-screen-list-and-prd.md`
- `05-information-architecture.md` 와 `07-supabase-data-model-draft.md`
- `08-auth-verification-moderation.md` 와 `07-supabase-data-model-draft.md`
- `11-roadmap.md` 와 `TASKS.md`

### C. 전공군 전략의 현실성

다음 질문을 던져라.

- 4개 전공군 동시 시작이 실제로 밀도 확보에 도움이 되는 구조인가
- 통합 홈 / 통합 모집 / 전공군 필터의 조합이 충분히 설득력 있는가
- 전공군이 4개라서 오히려 복잡도만 커지는 지점은 없는가

### D. 운영 리스크

다음 항목을 보라.

- 학생증 수동 인증 운영 부담
- 허위 모집/도배/분쟁 대응 가능성
- 학교 보드의 악용 가능성
- 익명성과 인증의 균형이 실제로 유지되는지

## 원하는 응답 형식

반드시 아래 형식으로 답해라.

### 1. 핵심 발견사항

- 심각도 순서대로 정리
- 각 항목마다 관련 파일명을 명시
- 가능하면 왜 문제가 되는지 짧고 명확하게 설명

예시 형식:

- 높음: `docs/02-mvp-scope.md`, `docs/09-screen-list-and-prd.md`
  - MVP 제외라고 해놓고 사실상 구현 전제가 들어가 있어 범위가 흔들린다.

### 2. 문서 간 충돌 여부

- 충돌 있음 / 없음
- 있다면 어떤 문서와 어떤 문서가 어떻게 충돌하는지 구체적으로

### 3. 빠진 항목

- 지금 문서 세트에 꼭 필요한데 빠진 것
- 우선순위가 잘못 잡힌 것

### 4. 과한 항목

- MVP에서 빼는 게 더 나은 항목
- 너무 이른 시점에 고려된 구조

### 5. 추천 수정안

- "당장 고쳐야 하는 것"
- "나중에 검토해도 되는 것"

## 리뷰 원칙

- 칭찬보다 문제 발견과 위험 지적을 우선해라.
- 애매하면 보수적으로 판단해라.
- 뜬구름 잡는 평가 대신, 실제 PM/기획/개발 관점에서 실행 가능성을 평가해라.
- "좋아 보인다" 같은 모호한 말 대신, 왜 괜찮은지 또는 왜 위험한지 구체적으로 써라.
- 기능 추가 제안은 최소화하고, 오히려 범위를 줄이는 쪽을 우선 제안해라.
